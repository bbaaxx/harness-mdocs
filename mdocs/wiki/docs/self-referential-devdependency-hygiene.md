# Self-referential devDependency hygiene

**Lesson:** Never list your own package name in its own `devDependencies`.

`harness-mdocs` (the package being published) was listed as `"harness-mdocs": "^0.1.1"` in its own `devDependencies`. A self-reference resolves to whatever version the registry holds for that name — a stale old copy lands in `node_modules` and can shadow the local `src/`/`dist/` build, causing phantom import resolution and release-pack:check drift.

## Fix
Removed the entry. Confirmed no source/test file imports the package by its published name (all imports use relative `src/`/`dist/` paths or the `.agents/bin/mdocs` runner). Verified clean: `npm run build`, `npm run release:check` (quality + pack:check) all pass.

## Check going forward
- `devDependencies` must not contain the package's own `name`.
- `grep -rn "from ['\"]<pkgname>" src/ test/` should return nothing for the published name.
- If a tool needs to exercise the published CLI, invoke the local build (`./dist/cli/index.js` or `.agents/bin/mdocs`), not the npm-resolved package.
