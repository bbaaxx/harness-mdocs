# wiki.ingest command + withLock promotion (G2b)

`wiki.ingest` (0.5.0) batches caller-supplied wiki operations into one isolated multi-file write. It composes existing `wiki.*` primitives + the G2a `updateOverviewSection`/`appendLog` helpers under `withLock`.

## Command
`wiki.ingest { operations: WikiIngestOp[], note? }` (registry: `src/core/commands/registry.ts`). Exposed as MCP tool `mdocs_ingest` and via CLI `mdocs command wiki.ingest --json '{...}'`.

### Op types
- `createPage` { category, id, title, content, tags?, relatedInitiatives?, lifecycle?, knowledgeType?, confidence? }
- `updatePage` { category, id, content?, lifecycle?, tags?, relatedInitiatives? } (errors `not found` if absent)
- `updateOverviewSection` { section, body } (no-op skip outside dir-v2, not an error)
- `appendLog` { entry } (no-op skip outside dir-v2, not an error)
- `link` { initiativeId, wikiSlug }

Returns `{ success, applied, operations[], changedFiles[], note }` — **caller data + structural metadata only**.

## Invariants
1. **No auto-prose.** Ingest records + applies ONLY what the caller supplies. The agent authors all text. No synthesis anywhere.
2. **Isolation, not rollback.** The whole batch runs under `withLock(mdocsRoot, 'wiki-ingest')`; each op is individually try/caught so one failure records an error but does not abort the rest. There is no transactional rollback.
3. **index.md byte-stable under external owner.** Ingest routes through `wiki.create`/helpers that no-op the canonical index when `wikiIndexOwner !== 'harness'`.

## withLock promotion
`withLock` moved from `src/surfaces/claude-code/lock.ts` to `src/core/lock.ts` (core is the right layer — registry is core). The surface file now re-exports it (`export { withLock } from '../../core/lock'`), so existing importers (`hooks.ts`, `surfaces/claude-code/index.ts`) are unchanged. `core/index.ts` exports it too.

The lock is mkdir-based (POSIX+Windows atomic), advisory, best-effort: on timeout the caller skips rather than blocks. Enforcement never hinges on it.
