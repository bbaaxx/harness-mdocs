# Consumer layering

harness-mdocs is generic. Real consumers layer workspace-specific conventions OVER it without conflicting with the plugin's hooks, skills, or CLAUDE.md snippet. Full guide: `docs/consumer-layering.md` in the repo.

## Key points
- **Self-contained mdocs/** — mdocs memory lives under `mdocs/`. Consumer docs/knowledge live in sibling dirs. Tool routing unchanged; no clobbering.
- **External task list coexistence** — mdocs initiatives are durable memory; an external task/issue source is the incoming stream. They coexist without conflict.
- **SessionStart composition** — Claude Code runs ALL matching SessionStart hooks and concatenates their `additionalContext`. Consumers can add their own SessionStart hook for workspace orientation alongside the G1 orientation hook (see [[sessionstart-orientation-hook]]). Keep each compact; ordering is not guaranteed.
- **CLAUDE.md snippet is additive** — `assets/templates/claude-md-snippet.md` is composed INTO a project's CLAUDE.md, not replaced. Append workspace sections after it; don't edit the snippet's tool-routing lines; re-merge on upgrade.

## Rule
Composition is EXTERNAL — consumers add their own hooks, CLAUDE.md sections, and sibling dirs. Never instruct consumers to edit package internals (src/, dist/, managers).
