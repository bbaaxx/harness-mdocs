# pi Surface

The pi surface (`src/surfaces/pi/`) exposes harness-mdocs as a [pi](https://pi.dev) package extension. It is a Tier 3 surface (full host-level enforcement) alongside OpenCode and Claude Code.

## Entry point

`package.json#pi` declares:
- `extensions`: `./dist/surfaces/pi/extension.js` ŌĆö compiled factory (run `npm run build` before `pi -e ./` from a git/local checkout).
- `skills`: `./src/surfaces/pi/assets/skills` ŌĆö three `mdocs-*` skills.

`./pi` package export exposes the capability table (`piSurface`), adapter, tools, and helpers for programmatic use.

## Extension factory (`extension.ts`)

Default export receives `ExtensionAPI` (type-only import ŌĆö no runtime value import from pi). Resolves the project root via the shared `resolveProjectRoot(process.cwd())`, builds one `MdocsCore` per load, and calls `lifecycle.ensureInitialized()`. pi reloads extensions on session switch/reload, so state refreshes naturally.

Registers four event handlers and 13 custom tools. Every handler is fail-open (try/catch, never throws ŌĆö a thrown handler can wedge the agent).

## Event mapping

- `tool_call` ŌåÆ enforcement gate. Calls `workflow.canExecuteTool(toolName, input)`; returns `{ block: true, reason }` when blocked. Blocks write/edit before PLAN; `./mdocs/` always allowed; bash audited not gated.
- `tool_result` ŌåÆ audit + progress. Appends a `tool` audit event (unlocked, safe under parallel tools). In full mode appends a progress-log entry under `withLock` when step !== IDLE; skipped in `metadata-only` mode so a thin consumer `_status.md` is not injected with a Progress Log section.
- `before_agent_start` ŌåÆ appends a compact (~1 KB) orientation banner to the system prompt each turn via `formatOrientationBanner(core)` (consumes `sessionContext`).
- `session_start` ŌåÆ user-facing `ctx.ui.notify` via `formatOrientationNotify`, guarded by `ctx.hasUI`.

## Tools (`tools.ts`)

13 tools, each with `promptSnippet` + `promptGuidelines` and a TypeBox schema: `mdocs`, `mdocs_init`, `mdocs_status`, `mdocs_validate`, `mdocs_search`, `mdocs_lookup`, `mdocs_dispatch`, `mdocs_ingest`, `mdocs_audit`, `mdocs_index_check`, `mdocs_resume`, `mdocs_advance`, `mdocs_reset`. Results wrapped via `toPiToolResult`; `{ error: string }` payloads surface `isError: true`. `mdocs_status`/`mdocs_resume` mirror the opencode rich shape (not the minimal `ops.*` shapes, which return empty for no-id listing).

## TypeBox peer dependency

`typebox` is an optional peer dep (`*`). pi bundles it and resolves it for extensions via jiti; do not bundle it. Tests stub `typebox` (`tests/stubs/typebox.js` + jest `moduleNameMapper`) because the real package ships ESM-only that ts-jest cannot parse; schemas are descriptive only (core never validates against them).

## Subagent dispatch

pi has no native subagent primitive. `mdocs_dispatch` returns an assembled context bundle carried forward manually. A future `/mdocs-subagent` command (`ctx.newSession`) is intentionally out of scope.

## Translation

pi built-in tool names are already lowercase and args use core camelCase keys, so `translate.ts` is a typed identity/passthrough (seam for future divergence), unlike the Claude Code surface which must map PascalCase/snake_case.

## Validation

Smoke-tested with `pi -e ./`: `mdocs_status` returns real workflow state, all three `mdocs-*` skills appear, and `write` is blocked at UNDERSTAND with the correct reason. Build/test/coverage/mdocs:validate/pack:check green; `dist/surfaces/pi` + `src/surfaces/pi` in the tarball.
