# Claude Code project MCP lives in .mcp.json

## Rule
For manual (non-plugin) installs, Claude Code reads project-scoped config from TWO different files:

- **`.mcp.json`** (repo root) → MCP servers. This is the only place project MCP servers are registered.
- **`.claude/settings.json`** → hooks, permissions, env. The `mcpServers` key here is IGNORED for project scope.

Putting `mcpServers` inside `.claude/settings.json` silently fails: the server never connects, `claude mcp list` shows nothing, and no error is raised. Discovered while dogfooding harness-mdocs's own claude-code surface.\n
## Correct manual install
```
.mcp.json                         -> { mcpServers: { mdocs: {...} } }
.claude/settings.json             -> { hooks: { PreToolUse, PostToolUse } }
.claude/skills/mdocs-*            -> copied from assets/skills/
CLAUDE.md                         -> mdocs snippet from assets/templates/
```
Restart Claude Code after creating both files — MCP servers and hooks load at startup.

## Exception: the plugin manifest
The INSTALLED-PLUGIN route is different: `src/surfaces/claude-code/plugin/.claude-plugin/plugin.json` legitimately bundles `mcpServers` + `hooks` together, and that works because Claude Code reads plugin manifests as a unit. The split applies only to hand-edited project config.

## Templates
`src/surfaces/claude-code/assets/templates/` now ships two files: `mcp.json` (server) and `settings-patch.json` (hooks only). Previously a single settings-patch.json held both, which caused the silent-disconnect bug.

## Path style
Absolute `${workspaceFolder}/node_modules/harness-mdocs/dist/...` paths are used in templates. For a repo-local dogfood (source IS the repo), absolute paths to the local `dist/` are more robust than `${workspaceFolder}`, which Claude Code may not expand in hook commands (unexpanded → hook file-not-found → fails open → no enforcement, silently).
