# Project-root resolution

All mdocs surfaces (MCP server, PreToolUse hook, PostToolUse hook) resolve the project root through ONE shared helper so the enforcement gate and the MCP server always operate on the same `mdocs/` root — even when `cwd` and `MDOCS_PROJECT_DIR` disagree.

## Helper
`resolveProjectRoot(cwd)` in `src/core/project-root.ts` (exported from `src/core/index.ts`).

## Precedence
1. **`MDOCS_PROJECT_DIR` env var** — if set and points at an existing directory. Explicit authoritative pin. Honored even before `mdocs/` is bootstrapped, so the `set env -> mdocs init` flow roots at the pinned dir (this deliberately does NOT require a `mdocs/` subdir to already exist).
2. **Walk up from `cwd`** — nearest ancestor (including `cwd`) that contains a `mdocs/` dir. Finds the owning project from a nested cwd (monorepo packages, subdirectories) when no env pin is set.
3. **Fallback to `cwd`** — preserves legacy `process.cwd()` behavior so a fresh `mdocs init` still roots at cwd.

## Why unified
Previously the MCP server used `MDOCS_PROJECT_DIR || process.cwd()` while the hooks used `payload.cwd || process.cwd()` (ignoring `MDOCS_PROJECT_DIR`). In multi-repo workspaces this made the gate and MCP operate on different roots. Every surface now passes its effective cwd through the same helper.

## Adding a new surface
Route the effective cwd through `resolveProjectRoot(...)` and pass the result as `projectDir` to `createMdocsCore`. Do not re-implement resolution inline. (G1's `session-start.ts` hook follows this pattern.)

## Non-breaking
With no `MDOCS_PROJECT_DIR` and a `cwd` that contains `mdocs/`, the result equals `cwd` (0.4.2 behavior).
