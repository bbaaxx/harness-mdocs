# Workflow state machine driver and bash gate

## Problem
The Claude Code surface advertised real PreToolUse enforcement (Write/Edit blocked before PLAN, destructive Bash before COMPLETE) but the workflow `currentStep` never moved: `WorkflowEngine.advance()` and `resumeAt()` had zero callers in production. `resume()` only set the active initiative; create/done only set/cleared it; the hooks read but never wrote `currentStep`. Since `canExecuteTool` returns true for every tool at IDLE, the gate was permanently inert during real work.

A second defect compounded it: the Bash branch used a non-destructive WHITELIST (`ls|cat|echo|...`) that omitted `npm`, `node`, `tsc`, `jest`, `npx`. So even if the state had advanced, builds and tests could not run at VERIFY (any non-whitelisted bash was treated as destructive, requiring COMPLETE), making VERIFY unreachable for real verification.

## Fix

### Driver (Bug 3)
- Registry command `workflow.advance { step }` validates the step against `STEPS` and calls `WorkflowEngine.advance()` (which enforces no-skip, no-back).
- CLI: `mdocs step <step>` (added to the usage string).
- MCP: dedicated `mdocs_advance` tool plus routing through the aggregate `mdocs` tool.
- `operations.advance()` thin wrapper for reuse across surfaces.

### Bash gate (Bug 4)
Switched from a safe-list to a destructive BLACKLIST so build/test tooling runs at VERIFY:
`rm|rmdir|mv`, `cp -f`, `git commit|push|reset|clean|rm`, `git checkout --|.`, `npm publish`, `mkfs|dd|shred`.

## Decision: redirects are NOT gated
File-overwrite redirection (`>`, `>>`) was initially included as destructive but produced unavoidable false positives: echo labels like `PLAN->COMPLETE`, shell arrows, and `2>&1` merges all matched, locking out legitimate work (and at one point locking the session out of its own rebuild). Regex cannot reliably distinguish a real redirect target from incidental `>` in a command string, so detection was dropped entirely. The documented destructive set above is what the gate enforces.

## Follow-ups
- After COMPLETE there is no command to reset to IDLE; the state file must be edited directly (or a reset command added).
- The Codex surface still has no advance path (advisory only); if Codex adopts enforcement it will need the same driver.

## Dogfood verification
Drove a real initiative IDLE through VERIFY via `mdocs step`; `npm run build`, `npm test` (292), and `mdocs validate` all ran at VERIFY, then advanced REPORT to COMPLETE. Enforcement binds for the first time under MCP/CLI-driven work.
